# Editing this site — what to change where

This repo (`knights-site` → GitHub `knightsontour/knights-around-the-world`) is the public
"Knights on Tour" travel blog. It builds on **Cloudflare Pages** — any push to `main` triggers a
rebuild and the live site updates in about a minute.

There are **two sources of truth**, and which one you edit depends on *what* you're changing.

## TL;DR

| What you want to change | Edit here | How it goes live |
|---|---|---|
| Chapter **text / photos** | The **Obsidian vault** note | Hit **⌘⇧P** in Obsidian |
| Trip **name, subtitle, cover, intro** (`_index.md`) | **This repo** | `git push` (manual) |
| **Menu, theme, site title, config** (`hugo.toml`, `layouts/`) | **This repo** | `git push` (manual) |

## The rule

- **Chapter content is owned by the vault.** Each chapter under `content/travel/<trip>/*.md` is
  *generated* from a note in the Obsidian vault by the `⌘⇧P` pipeline. **Do not hand-edit chapter
  files in this repo** — the next time that note is published with `⌘⇧P`, the converter regenerates
  the file from the vault and your manual change is overwritten. Edit the vault note instead.

- **Structure & config are owned by this repo.** These files are **not** generated by the converter,
  so editing them here is correct and safe — they won't be overwritten:
  - `hugo.toml` — site title, menu / Trips dropdown, theme settings, `buildFuture`, `baseURL`
  - `content/travel/<trip>/_index.md` — each trip's display **title**, **subtitle**, **cover image**
    (`image:`), and intro text
  - `layouts/` — homepage tiles, post layout, the `travelmap` shortcode, etc.
  - `static/` — assets

  These only go live when you commit and push them yourself.

## ⌘⇧P does NOT publish manual repo edits

`⌘⇧P` runs from Obsidian: it converts the *vault note* you're on and pushes **just that one file**.
It has no awareness of edits you make directly in this repo. So after editing `hugo.toml`, an
`_index.md`, or anything under `layouts/`, publish them yourself:

```bash
cd ~/GitHub/knights-site
git add -A
git commit -m "describe your change"
git push          # Cloudflare rebuilds in ~1 min
```

## Editing on GitHub.com directly is fine too

Any commit to `main` deploys automatically — `git push`, the `⌘⇧P` pipeline, **or** editing a file
in the GitHub web UI. So tweaking an `_index.md` or `hugo.toml` on github.com works and goes live on
its own.

**Just remember to sync your local clone afterwards** so it doesn't fall behind GitHub (otherwise
your next local `git push` hits a conflict):

```bash
cd ~/GitHub/knights-site
git pull          # pull down anything you committed on GitHub
```

## Adding a brand-new trip (one-time wiring)

The `⌘⇧P` converter creates chapter files but **not** a trip's landing page or menu entry. The first
time a new trip is published, do this once in the repo:

1. Create `content/travel/<slug>/_index.md` with `title`, `subtitle`, and an intro. **Without it the
   trip's pages 404** and it won't tile on the homepage. (Optionally add `image: "/assets/<file>"`
   to set the tile cover; otherwise the first chapter's first photo is used.)
2. Add a child entry under the `trips` menu in `hugo.toml` (`parent = "trips"`, `name`, `url`,
   next `weight`).
3. `git commit && git push`.

If a trip's chapters are dated in the **future** (e.g. written ahead of the trip), make sure
`buildFuture = true` is set in `hugo.toml` (it is) — otherwise Hugo skips them and they 404.
